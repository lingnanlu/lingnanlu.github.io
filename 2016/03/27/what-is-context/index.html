什么是Context

发表于 2016-03-27

在写程序时, 经常遇到Context, 但一直不懂, 利用这篇文章总结一个

先看一个涉及到的几个类的UML

最关键的就是Context, 该类的官方文档说明

Interface to global information about an application environment. This is an abstract class whose implementation is provided by the Android system. It allows access to application-specific resources and classes, as well as up-calls for application-level operations such as launching activities, broadcasting and receiving intents, etc.

即通过该接口可以访问应用级的信息, 什么是应用级的信息呢? 就是在四大组件之上, 不隶属于组件的信息和动作, 如启动新的Service, 启动新的Activity, 获得应用的包名等等, 这些动作和信息显然不属于某一组件.

通过其接口名称可以感知一二

//应用级的动作
bindService(Intent service, ServiceConnection conn, int flags)
registerReceiver(BroadcastReceiver receiver, IntentFilter filter)
sendBroadcast(Intent intent)
startActivity(Intent intent)

//应用级的信息
getCacheDir()
getExternalCacheDir()
getExternalFilesDir(String type)
getMainLooper()
getPackageName()
getResources()
getSharedPreferences(String name, int mode)
getTheme()

ContextImpl就是其具体实现

ContextWrapper包含一个ContextImpl, 所有的操作都是委托给ContextImpl对象, 那么其作用是什么? 官方文档说明如下

Proxying implementation of Context that simply delegates all of its calls to another Context. Can be subclassed to modify behavior without changing the original Context.

这里利用了装饰器模式, 即可以在不改变ContextImpl以及相应接口的情况下为其添加功能, 如原始的函数都没有Log功能, 装饰器类可以为函数添加Log功能.

ContextWrapper需要由更具体的装饰器类继承

ContextThemeWrapper, 一个更具体的装饰器类, 添加了theme相关的功能.

由图中可见, Activity继承了ContextThemeWrapper, 因为Activty有界面.而Service没有继承ContextWrapper, 因为Service不需要界面.

还有一个类是Application, 该类的对象就代表着一个应用, 所以该类是单例模式, 一个应用只有一个对象.

关于Activity, Service, Application对象生成的时机以及与一个ContextImpl对象绑定的时机, 根据参考文章中的源代码分析, 以下给出结论:

  1. 整个应用就一个Application对象, 该对象关联一个ContextImpl对象
  2. 每个Activity和Service都关联一个ContextImpl对象

由上可知, 在整个应用中, 有如下数据关系

ContextImpl实例个数 = Service个数 + Activity个数 + 1

这里有个疑问, 为什么需要多个ContextImpl实例? 一个不行么?

网上找到的答案是:

ContextImp是一种轻量级类，而PackageInfo才是真正重量级的类. 而一个App里的
所有ContextIml实例，都对应同一个packageInfo对象。

这一点继续留有疑问

两者返回的其实是一个对象, 只是返回值不同, 一个为Application, 一个为Context
