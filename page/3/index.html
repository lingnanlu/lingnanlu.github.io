Netty Executor UML

发表于 2016-08-09

Netty中和Executor相关的有很多类, 如EventExecutor, EventExecutorGroup, EventLoop, NioEventLoop等等, 理清它们之间的关系很重要, 将它们画成UML图如下所示

浅蓝:接口-核心概念
橘黄:抽象类-模板
深蓝:实现

对该图结构的理解关系到各个类的理解. 关于该图, 有以下几个值得注意的地方.

  1. 接口代表着核心概念， 所以理解接口最重要
  2. 可以把接口看作是一种功能, 接口之间的继承关系可以看做是功能的不断扩展, 越往下层走， 功能越强, 比如说EventExecutorGroup就是对ScheduledExecutorService的功能进行了扩展.
  3. 每一层都提供模板类来提供对接口的默认实现， 这是为了减少实现类的代码
  4. 每一层的模板类继承上一层的模板类是了为复用上一层接口的默认实现， 如果不继承也是可以的， 只是还要再提供一次上层的默认实现. 而实现本层接口是为了提供本层扩展功能(相对于上一层来说)的默认实现
  5. 整个图可以看作是一个层次结构， 需要哪一层的功能就扩展那一层的抽象类即可.

结合本图和文档中各个类的说明, 就可以对类之间的关系有一个比较清晰的认识了. 而知道了类与类之间的区别, 则对类本身的含义也就可以更加明确了.

Java容器总结

框架图

fast-fail原理

当多个线程对同一个集合的内容进行操作时，就可能会产生fail-fast事件。
例如：当某一个线程A通过iterator去遍历某集合的过程中，若该集合的内容被其他线程所改变了；那么线程A访问集合时，就会抛出ConcurrentModificationException异常，产生fail-fast事件。

原理见Java 集合系列04之 fail-fast总结

List

1 ArrayList

  1. 容量不足时，新扩容的大小为旧容量的一半
  2. 实际上利用的一个数组去保存数据的

  3. 不同遍历方式

    • get(index)方式最简单最高效， 就是直接返回底层数组的index元素，而没有fast-fail检查
    • 而使用iterator会涉及到fast-fail检查， 即多个线程并发去读写list时， 抛出concurrencymodify异常， 所以效率低

2 CopyOnWriteArrayList

  1. 顾名思义，在对List写时， 所写的已不是原数组，而是在一个新的数组在进行操作
  2. 由源码可知， 如果有多个线程并发读写， 那么读到的数组是原数组， 所以会出现数据不一致的情况， 即读到的数据并不是最新的。但因为最终读（在所有写完成后）可以读到最新数据， 这就是具有最终一致性， 但不具有实时一致性
  3. 适用于读多写少（读的时候不加锁）， 且不要求实时一致性的场合
  4. 在写的时候会进行加锁， 所以， 即使有多个写线程， 也是只有一个副本。
  5. 因为写的时候有一个副本， 所以内存还是占用多那么一点
  6. 其它CopyOnWrite系列原理与此相同

参考文章

Java中的CopyOnWrite容器

3 LinkedList

  1. 实际上是一个双向列表
  2. 其依然支持随机访问， 它就是通过一个计数索引值来实现的。例如，当我们调用get(int location)时，首先会比较“location”和“双向链表长度的1/2”；若前者大，则从链表头开始往后查找，直到location位置；否则，从链表末尾开始先前查找，直到location位置
  3. 依然是fast-fail机制
  4. 由随机访问的原理可知， 当遍历LinkedList是， 随机访问的效率是很低的，所以使用Iterator比较好

4 Vector

和ArrayList类似， 但是是线程安全的.

因为是默认是线程安全的， 所以get，set方法都要加锁， 即使是在非并发环境下。

所以，新的设计是使用ArrayList, 如果在并发环境中使用， 则利用synchronizedList返回其并发版本。

5 Stack

Stack是继承Vector的， 所以Vector的缺点它都有， 这也是不推荐使用的原因，可利用LinkedList代替

Map

1 HashMap

  1. HashMap是非线程安全的，一样实现了fast-fail机制
  2. 使用Collections.sy…Map方法返回其线程安全版本，该安全版本性能不好，因为锁的是整个Map, 性能更好的版本是CocurrencyHashMap，这个使用的是分段锁，锁的不是整个Map
  3. HashTable是线程安全的, 但同样性能不好
  4. HashMap中loadfactor的含义

  5. HashMap提供三种视图

    • 所有条目entrySet():集合
    • 所有键keySet():集合
    • 所有值values():Collection

  6. put, get的复杂度为O(1)

2 HashTable

和HashMap类似， 就是线程安全的， 性能低

3 TreeMap

  1. 利用红黑树来保存， 因为红黑树也是用来加快检索的， 所以和Hash的功能类似
  2. 因为红黑树也是二叉搜索树， 对其进行中序遍历， 就可以得到其有序序列， 其保存的键是有序的， 所以当对其进行迭代时，键是有序的
  3. 因为是红黑树，其put,get操作复杂度是O(logN)
  4. 同样，非同步，fast-fail

4 WeakHashMap(TODO, 以及WeakReference的用途)

  1. 键是弱键，值不是弱值
  2. 当弱键引用的对象被回收后，会将该弱键放入ReferenceQueue中，当下一次操作WeakHashMap时，会根据ReferenceQueue中的弱键删除相关entry
  3. 同样，非同步 fast-fail
  4. 用途，好像是用来保存某些元信息(TODO）

5 LinkedHashMap

保存着插入顺序，迭代时的顺序就是插入顺序。继承自HashMap, 非线程安全

Set

Set是基于Map实现的，其中

  1. HashSet是基于HashMap
  2. TreeSet是通过TreeMap

1 HashSet

  1. HashSet是基于HashMap
  2. 非同步，fast-fail
  3. 怎样实现没有重复元素呢？实际上是利用了HashMap中键不可能重复的性质，也就是说，只使用了HashMap中的键，而值是一个Object对象
  4. 其它性质与HashMap类似，如查找性质等

2 TreeSet

参照HashSet的4条与TreeMap的性质。

理清概念-框架,DIP(依赖倒置),IoC(控制反转),DI(依赖注入)

发表于 2016-08-07

本文目的有二

  1. 讲解以上各个概念的含义
  2. 各个概念之间的关系与区别

一 框架

框架是什么？

网上有不同的诠释， 本文给出以下解释（只是自己的见解， 不一定正确）

框架是解决一类问题的一种抽象， 可用来解决特殊性问题。

以上的解释还是很抽象， 下面举一个简单的例子，从中可以看出框架是如何产生的， 进而理解其含义。

假如说， 我们现在想解决这样一个问题

假如数据库保存着一个电影列表， 我们想将某导演导的电影输出到控制台

代码可以这样写

现在， 又有新的需求了

假如从网络得到一个电影列表， 我们想将某导演导的电影输出到控制台

代码不写了， 应该和前面是类似的。

于是， 每有一个新的需求， 那么， 除了添加新的Finder外(这是不可避免的)， 还要去修改MovieListPrinter。

有人说， 虽然修改MovieListPrinter， 但每次只修改了一行， 大部分的代码还是可以重用的， 其实不然

  1. 本例很简单， 所以只修改了一行
  2. 重用的概念是针对某一程序中的实体来说，最小实体可用是一个函数， 大一点是一个类， 再大一点是一个包， 而且重用要求不对实体进行修改，而上面的重用只是重用了几行代码， 这和从其它地方复制粘贴几行代码有什么区别， 难道复制粘贴也叫重用？

所以说， 当问题变化时， 我们的代码就相当于推到重来。之前的代码一点也不能重用。

如果仔细考虑一下以上三个问题， 可以发现他们是一组类似的问题， 抽象一下可以得到如下表述

从某个地方得到一个电影列表， 某导演导的电影输出到控制台

如果我们对照着这个更一般的问题来写代码， 就是这样子的了。

可以看到， MovieListPrinter中没有再和具体finder相关的代码了， 所以说MovieListPrinter就是可以重用的, 只要我们针对不同的具体问题传入不同的MovieFinder即可， 而MovieListPrinter本身则一点也不需要修改， 所以可以是重用的。

现在， MovieListPrinter就可以称为了一最最简单的框架了， 它解决的是如下问题

从某个地方得到一个电影列表， 某导演导的电影输出到控制台

再对比开始的定义， 应该明白了不少吧。

框架是解决一类问题的一种抽象， 可用来解决特殊性问题。

二 DIP(依赖倒置)

依赖倒置是软件设计模式的一种原则， 它是什么含义呢？

其解释出现在《敏捷软件开发》的第11章

  1. 高层模块不应该依赖于低层模块， 二者都应该依赖于抽象
  2. 抽象不应该依赖于细节，细节应该依赖于抽象。

这里提到的高层模块， 低层模块， 抽象， 细节到底指的是什么呢， 依赖倒置又是什么含义呢？

书中的一句话指明了回答以上问题的线索

该原则是框架设计的核心原则

而之前已经介绍了框架， 并设计了一个简单的框架， 也就是说， 在设计的过程中， 已经不知不觉的应用了该原则。 那么结束以上我们的设计过程， 就可以解释以上几个概念的含义了。

高层模块：对应的是MovieListPrinter, 它表示的是解决问题的一般思路, 它表示的是策略

低层模块：对应的是每一种MovieFinder， 其只能解决某一具体问题

从各种MovieFinder中抽象出来的MovieFinder接口就是抽象， 而具体的MovieFinder就是细节

在设计框架之前

MovieListPrinter（高层模块）依赖于各种MovieFinder(低层模块)，

使用了框架后

MovieListPrinter(高层模块)依赖于MovieFinder接口（抽象）， 同时， MovieFinder实现(细节）也依赖于MovieFinder接口（抽象）（这不就是开头的两句话吗）

对比框架使用前后， 依赖关系发生了倒置（更形象的可以参考《敏捷软件开发》的第11章)

所以不知不觉的， 在我们设计MovieListPrinter框架中就使用了DIP原则。

注：高层模块和低层模块并不是绝对的， 在一个复杂的系统中可以有好多层， 某一层对上来说是低层模块， 对下来说又可以是高层模块

三 IoC和DI

在谈这两个概念之前， 先回到开始的MovieListPrinter框架。

假如应用程序中要使用这个框架，那么我们需要在其构造函数中传入一个具体的MovieFinder. 如我们的程序要从文件中读入电影列表，那么传入一个FileBasedMovieFinder即可。

现在需求改了， 要从网络去读了， 那么需要修改应用程序。

如果修改一个地方还好， 假如要有多个地方需要修改呢？ 假如这多个地方分布在几百的文件中呢？

还有一种情况，现在我们的框架很小， 复杂的框架可能有复杂的依赖关系， 如A->B->C->…..->G

可以， 如果我们要在应用程序中去人工的去处理这种依赖， 则很麻烦， 容易出错。

如果有以下这样一种方法就好了。

不是通过编程来注入某些依赖， 而是通知声明式来注入。 框架使用者只需要声明一个A需要B,那么就自动的为A注入一个B

而以上这种机制实际上是有的。

以下举两个例子（其实现来自于本人猜测， 不保证正确）

  1. Spring：在XML中声明依赖关系， 通过分析XML文件构建图来表示依赖关系，由Spring来负责提供一个类所需要的依赖。
  2. Dagger: Android的依赖注入库， 利用注解来声明依赖关系， 同样是需要构建某种图

现在可以说一下这两个概念了。

IoC：是将由应用程序负责定位某种依赖转移到由某种外部机制来定位某种依赖（为什么叫反转呢？我也不知道）

DI: 上面说了由某种外部机制来定位， DI就是一种外部机制， 具体的例子见Spring和Dagger. 还有一种方式叫到Service Locator这是另一种定位某种依赖的机制

四 Spring

Spring的功能之一就是实现DI， 从这个意义来说， 其并不能算是一种框架， 当然如果把解决DI问题看作一般问题的话， 也可以称其为DI框架。

HashMap中loadfactor的含义

发表于 2016-08-07

JDK文档中提到

An instance of HashMap has two parameters that affect its performance: initial capacity and load factor. The capacity is the number of buckets in the hash table, and the initial capacity is simply the capacity at the time the hash table is created. The load factor is a measure of how full the hash table is allowed to get before its capacity is automatically increased. When the number of entries in the hash table exceeds the product of the load factor and the current capacity, the hash table is rehashed (that is, internal data structures are rebuilt) so that the hash table has approximately twice the number of buckets.

As a general rule, the default load factor (.75) offers a good tradeoff between time and space costs. Higher values decrease the space overhead but increase the lookup cost (reflected in most of the operations of the HashMap class, including get and put). The expected number of entries in the map and its load factor should be taken into account when setting its initial capacity, so as to minimize the number of rehash operations. If the initial capacity is greater than the maximum number of entries divided by the load factor, no rehash operations will ever occur.

注意其中

The load factor is a measure of how full the hash table is allowed to get before its capacity is automatically increased.

但这句话到底是什么含义呢？

要想理解以上这句话，需要了解有关HashMap的其它几个术语

  1. capacity：指桶的个数
  2. size：指Map中Entry的个数
  3. threshold：threshold = capacity * loadfactor, 含当size>=threshold时， 增加capacity为原来的2倍（参考源代码）
  4. loadfactor: 含义是什么呢？

不妨通过一个例子来了解loadfactor的含义

假设capacity=10

  1. 假设loadfactor=10, 由以上可知, 当size>=(threshold = capacity * loadfactor)时扩容， 即size>=100时扩容，
    而此时只有10个桶， 可以看出， 每个桶都是比较满的。但扩容的速度比较慢

  2. 假设loadfactor=0.1,则size=1时扩容， 而此时有10个桶， 大多数桶还是很空的。但扩容的速度比较快。

于是， 可以理解上文中那句话了。

结论

由上讨论可知

当loadfactor比较大时， 桶比较满， 空间占用比较少，扩容速率比较慢， rehash次数比较少, 但每个桶中元素比较多， 所以查找效率会低一些

当loadfactor比较少时， 桶比较空， 空间占用比较多，扩容速率比较快， rehash次数比较多， 查找效率会高一些

最终是在时间和空间中找到一个平衡
