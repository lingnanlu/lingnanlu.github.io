huffman树基础知识

发表于 2016-05-09

关于哈夫曼树, 关注其背后的抽象

  1. 路径: 根到某一个子结点的通路, 称为路径
  2. 路径长度: 通路上分支的个数为长度.
  3. 结点的权: 对于每一个结点, 有一个值, 称为结点的权
  4. 带权路径长度: 对于一个结点, 其根到该结点的路径长度与该结点权的乘积
  5. 树的带权路径长度, 所以叶子结点的带权路径长度之和, 注意是叶子结点

要了解以下抽象背后的含义

1. 结点的权的意义

叶子结点的权有实际意义, 非叶子结点的权只是构造树的过程中起辅助作用.

2. 带权路径长度的含义

主要关注的是叶子结点的带权路径长度的含义

3. 路径及路径长度含义

主要关注的是叶子结点路径及路径长度的含义

4. 树的带权路径长度的含义

一般是解决这样的一问题, 该问题中有一类实体, 每个实体有不同值, 为该实体再分配另一个值, 要求所有实体分配的值与该实体的值的乘积和最小.

典型应用:编码

实体为要编码的字符

实体的值为字符出现的频率

分配的值为该字符的编码

要求最编码长度最小

至于前缀问题，这个是由树的性质来保证的，而不是Huffman树独有的

哈夫曼树的构造

因为每次要找两个最小权值的元素, 所以可以利用最小堆, 不断的从堆顶取元素, 并将合并的元素再插入回堆中.

在Java中可以利用优先级队列来模拟堆的操作.

计算机是如何将一个画面显示到屏幕上的?

发表于 2016-05-09

涉及到三个部分

  1. GPU
  2. 内存
  3. 显示器

绘制过程

首先GPU将要显示的画面写入到内存的一块区域, 然后, 显示器将内存区域的内容读出, 并利用逐行扫描来显示该画面.

由以上叙述可知, 这是一个生产者消费者的模型,

  1. 生产者->GPU
  2. 消息者->显示器
  3. 产品->画面, 也称为帧

帧速率, 刷新率

帧速率: GPU每秒往内存中写的帧数
刷新率: 显示器每秒从内存中读出并逐行刷新显示出来的帧数

当两者相同时, 显示出来的画面是稳定的, 流畅的, 如图一

当帧速率大于刷新率时, 假如说帧速率是刷新率的2倍, 那么当显示器是读内存的下半部分时, 其实已被写入了下一帧的内容, 导致的结果就是一个画面上显示了两帧的东西, 这叫做画面撕裂.如图2所示

当刷新率大于帧速率时, 假如刷新率是帧速率的2倍, 这会导致同一帧显示两次, 其实, 如果一直这样也没啥, 画面给人感觉也是很流畅的, 但是如果一开始刷新率与帧速率相等, 之后帧速率下降, 这给人的感觉就是一个画面的显示时间相比之前变长了, 直观的感觉就是卡了(卡就是长期不变)

如何解决

撕裂问题

可以利用两个缓冲区, 一个为帧缓冲, 一个为后缓冲, GPU将帧写入后缓冲, 而显示器只从帧缓冲中读帧, 将读完一帧后, 通知GPU将后缓冲中的帧复制到帧缓冲中, 这样, GPU可以继续往后缓冲中写帧, 而显示器可以从帧缓冲中读帧. 因为是显示器读完帧后发信号通知GPU, 所以最终的画面的流畅度取决于刷新率. 如图三

卡顿问题

卡顿是由帧速率下降导致, 而帧速率下降是由于某一画面过于复杂, GPU计算生成该画面过长所致.

阻塞同步(互斥同步)与非阻塞同步

发表于 2016-05-09

阻塞同步

就是使用synchronized等进行的同步, 因为需要进行线程的阻塞与唤醒, 而这两个操作需要进行系统调用, 进行用户态与核心态的转换, 其性能影响较大.

这是一种悲观的加锁策略, 悲观指的是:总是认为只要不去做正确的同步措施, 那就肯定会出现问题, 无论共享数据是否真的会出现竞争.

非阻塞同步

即不使用synchronized等, 不会进行线程的阻塞与唤醒. 其实现往往依赖于乐观加锁策略.

那么什么是乐观加锁策略(Lock-free, 无锁同步)?

当一个线程要修改共享数据时, 假定其它线程不会修改共享数据, 所以对对共享数据进行读入和修改时, 并不需要加锁,只有将共享数据保存时, 需要作一个判断, 那就是, 在将新值保存时, 旧值有没有被修改过, 如果没有则说明没有其它线程修改过数据, 所以可以放心的保存(其实这样说不完全正确, 因为存在ABA的问题, 即其它线程将数据A修改为B后, 又修改回了A). 如果被修改过, 放弃保存, 从新读取数据, 重复此过程, 一直到可以保存为止. 可见该过程对共享数据的修改并没有利用加锁机制.

自旋锁.

自旋锁指的是一种获取锁的策略. 在互斥同步中, 如果一个线程试图去获取一个已被占用的锁, 会被阻塞, 我们知道, 阻塞的开销很大.

而自旋是指, 如果获取不了锁, 并不阻塞. 而是不断去尝试获取锁, 直到能获取锁.

那么现在问题来了, 如果一直尝试获取锁, 就会一直占用CPU, 而持有锁的线程得不到CPU就不会释放锁, 岂不是会造成一直等待?

其实, 自旋锁的使用是有一个前提的, 那就是有多个CPU, 可以同时运行多个线程, 那么以上问题就可以解决了.

而且自旋锁的出现是为了解决阻塞开销过大问题, 而这个开销过大只是相对来说, 相对于谁呢? 相对于线程的操作来说, 如果一个线程对共享数据的操作时间远远小于其阻塞的开销时间, 那么使用自旋锁是合适的.
如果阻塞的开销远小于对共享数据的操作时间, 即可以忽略不计, 那么使用阻塞也不会太影响性能.

而且, 如果另一个线程长期持有锁, 也不适合自旋, 因为会长期在CPU上忙等, 浪费CPU的资源. 此时可以设置个阈值, 当自旋一定次数后, 还得不到锁, 就阻塞挂起.

Walk顺序与Visit时机

发表于 2016-05-09

一个比喻

我们可以将树型结构比作一个小区, 小区中的房子都是一座座独立的别墅.

树的结点–>别墅

树的分支–>连接别墅的道路

Walk顺序

假设树所对应的小区的结构如下图所示(之所以添加null结点是为了下文中讨论Visit时机时方便, 同时也可以让代码更容易理解与简化)

假如张三在小区散步, 他的散步习惯是深度优先. 出发点是A, 则其经过的别墅顺序为

A->B->Null->D-Null->Null->C->Null->Null

而以上的顺序就是Walk顺序, 注意, 在经过某一别墅时, 他并没有进入到别墅中拜访别墅的主人.

由以上讨论可知, 对于一个小区, 如果散步习惯是深度优先, 其Walk顺序是确定的.

Visit时机

如果说, 当张三经过某一别墅时, 决定进入别墅拜访主人, 我们将拜访的抽象为Visit操作, 对于每一个别墅来说,Visit的时机, 有三种可能.

对于A, 对于A其时机可以是

时机1:进入B之前, 这个时机可以称为进入A结点的时机

时机2:从B返回

时机3:从C返回, 这个时机可以称为离开A结点的时机

当涉及到树的结构时, 可以根据不同的需要选择一个或多个时机, 对于每一个时机, 也可以进行不同的操作.
