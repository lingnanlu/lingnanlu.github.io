由一个实例了解AMS机制

发表于 2016-02-28

前言

本文通过分析一个具体的实例来了解AMS的机制,了解该机制中涉及到的实体,这些实体之间的关系,以及这些实体的作用的角色等.

该实例来自于Android系统在新进程中启动自定义服务过程（startService）的原理分析, 本文只是换种方式来对其进行重新组织和分析,并只关注AMS中的关键概念.

实例分析

该实例可分为三个阶段,以下分阶段进行分析.

1. Step 1 - Step 7:要启动服务的进程与ActivityManagerService通信

所做工作

  1. 启动一个新的进程
  2. 从Intent中取得Service的相关信息(如要运行的Service的类名)
  3. 将新进程的pid与Service的相关信息保存到ActivityManagerService中

涉及实体

  1. ActivityManagerProxy
  2. ActivityManagerService
  3. IActivityManager
  4. ContextImpl
  5. ActivityThread
  6. Context
  7. ContextWrapper
  8. ContextThemeWrapper
  9. ContextImpl
  10. Activity

以上类的之间的关系如下图所示

由图可知,当调用Activity的startService时,实际上调用的是ActivityManagerProxy,而从图中看出,IAcitityManager, ActivityManagerProxy, ActivityManagerService是典型的Binder机制结构, 所以最终调用的是ActivityManagerService.startService().

对于ActivityThread, 在Android系统中,每一个进程都用一个ActivityThread实例表示,每一个进程都与ActivityThread相关联.体现以上说法的是ActivityManagerService.startProcessLocked()中的如下代码:

int pid = Process.start("android.app.ActivityThread",  mSimpleProcessManagement ? app.processName : null, uid, uid,
                        gids, debugFlags, null);

可见,ActivityThread是由新的进程执行,而其main函数就是新进程程序的入口.

2. Step 8 - Step 11:由第一阶段生成的新进程与ActivityManagerService通信

所做工作

获取第一阶段中保存在ActivityManagerService中的Service信息(此时并未把该信息传到新进程中,只是查询出来)

涉及实体

  1. ActivityManagerProxy
  2. ActivityManagerService
  3. IActivityManager

3. Step 12 - Step 20:ActivityManagerService与新的进程进行通信

所做工作

  1. 将第二阶段中查询得到的信息发送回新的进程
  2. 由新的进程负责生成相应的Service实例

涉及实体

  1. IApplicationThread
  2. ApplicationThreadProxy
  3. ApplicationThread
  4. ActivityThread
  5. ActivityThread.H
  6. 自定义的Service

其中1, 2, 3也是一个典型的Binder结构,也是用来在进程间通信, 此时ActivityThread保有一个ApplicationThread的对象,ApplicationThread是IApplicationThread的本地接口,那么远端接口在哪里呢? 远端接口在ActivityManagerService那里,是由第二阶段的ActivityManagerProxy.attachApplication(IApplicationThread)将IApplicationThread的远端接口发送给ActivityManagerService的.

由本地接口与远端接口所处的进程可以知道进程通信的方向是由ActivityManagerProxy到新的进程.而1, 2, 3的存在就是为了支持这一方向的进程间通信的Binder机制.

当ActivityManagerService将其第二阶段中保存的Service信息传给新的进程之后,由新的进程负责创建Service的实例.在新进程中,会通过Looper消息机制来创建Service实例.而Looper消息机制的建立正是在第一阶段中ActivityThread.main()函数中建立的.

创建完Service实例后,会调用其onCreate()方法.

总结

由以上分析可知,上面的进程通信有两个方向,一个是进程到ActivityManagerService, 另一个方向是ActivityManagerService到进程. 于是就有两个Binder机制与之相对应.

对于进程到ActivityManagerService

  1. ActivityManagerProxy
  2. ActivityManagerService
  3. IActivityManager

对于ActivityManagerService到进程

  1. IApplicationThread
  2. ApplicationThreadProxy
  3. ApplicationThread

查看IActivityManager接口,发现有如下方法等

startActivity

stopActivity

startService

stopService

bindService

可见,虽然名字是ActivityManager,但其功能却是对Android四大组件进行管理.这里名字有一定的误导

而IApplicationThread也有如下这些接口等

schedulePauseActivity

scheduleStopActivity

scheduleCreateService

其实,IApplicationThread与IActivityManager都是对四大组件进行管理,而在其中扮演不同的角色罢了.

另外,ActivityThread也并非只和Activity相关,它与一个进程相关联,或者说它就代表了一个进程,与IActivityManager, IApplicationThread配合,对activity, broadcast,service等进行管理

所以在新的进程中启动Service或Activity的过程的关键点如下:

  1. 利用两套Binder机制信息交换
  2. 无论是Service还是Activity,都会执行ActivityThread.main方法, 从而在新的进程中可以处理UI的重绘.

compileSdkVersion minSdkVersion targetSdkVersion SupportLibrary含义与设置策略

发表于 2016-02-25

如何让app支持更多的系统版本,并且能利用最新系统的特性(如android 6.0 的运行时权限检查?)

在开发app时有几个与次相关的概念: compileSdkVersion, minSdkVersion, targetSdkVersion, Support Library.

minSdkVersion:

从字面意思去理解就是该app所支持的最小的sdk,如果系统的sdk小于该值,则系统不能安装该app.

另一方面,该值也限制了该app所能使用的api,如minSdkVersion=13, 那么高于13的api就不能使用,因为如果使用了的话,在系统sdk=13的平台上该app就无法运行,与其声称的支持最小13相背.

那么,如何能即不违背其声明,又想在sdk大于13的平台上使用新的api呢?可以使用如下的方式:

if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.HONEYCOMB) {
    ActionBar actionBar = getActionBar();
    actionBar.setDisplayHomeAsUpEnabled(true);
}

targetSdkVersion:

该值表示该app是为哪一个版本的api构建的,既然是专为某一个版本的api进行构建的,其潜在含义就是该app在该api上表现的最好? 那么,什么是表现的最好呢? 即可以利用该版本的最新特征,如targetSdkVersion=23,就可以利用android 6.0 的运行时权限检查特征.如果targetSdkVersion小于22, 那么该app同样可以在android 6.0设备上运行,但是不能利用其运行时权限检查特征(这里不能利用特征,但并非不能利用其新的api,能否使用新的api要看compileSdkVersion).

如果targetSdkVersion高于运行环境的版本,那么依然能利用运行环境的特征,只有在低于时,运行环境会以兼容模式来运行程序,所以不能利用其最新特征.

由以上可以,targetSdkVersion只影响app是否能利用最新平台的行为变化,而不影响能使用的api,所以在新平台发布时,为了使app利用新平台最新的行为变化,需要修改targetSdkVersion为最新平台的version.

compileSdkVersion:

这个决定了构建app时,所使用的android.jar版本,可以把android.jar当做一个java第三方库,在构建程序时,当然需要指定第三方库的版本.同时不同的版本提供的api不同,所以该值限制了你能使用的api.

另外,由于android是向后兼容的,新版的sdk包含着旧版sdk中的api,所以如果只使用旧版的api,那么使用新版的sdk构建出来的app依然可以在低于新版sdk的平台上运行.

所以一般指定compileSdkVersion为最新的sdk版本.

如何既想支持老的版本,又想使用新的api呢?

支持老的版本需要指定尽可以小的minSdkVersion, 而又使用新的api则必须指定较大的compileSdkVersion,但小的minSdkVersion限制了api的使用,所以在老版本中支持一项在新版本中的特征可以这个去写,如支持actionbar:

if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.HONEYCOMB) {
    //使用native api
    ActionBar actionBar = getActionBar();
    actionBar.setDisplayHomeAsUpEnabled(true);
} else {
    //使用老版本的api来模拟其新特征.
}

而Support Library正是对以上这种代码结构做了封装,判断平台的sdk版本来决定是否使用原来的api还是使用老版本的进行模拟.

所以利用Support Library就可以解决以上提出的问题.

结论

一般开发应用时,三个值的关系如下

minSdkVersion (lowest possible) <= targetSdkVersion == compileSdkVersion (latest SDK)

并且使用Support Library来实现新SDK才有的特性.

android Binder相关知识

发表于 2016-02-25

背景知识

  1. JNI: JAVA基础之理解JNI原理
  2. Stub在软件开发中的含义
  3. client-server模型:见CSAPP中网络编程一章
  4. 事务: 事务就是client-server模型中,client与server的一次通信过程的抽象表述, 详见CSAPP中网络编程一章
  5. 远程过程调用: 一个进程调用另一个进程中对象的方法,这就称为远程过程调用,其中另一个进程中对象的方法称为远端方法.其实现机制利用的是进程间通信的原理,比如说A进程通过socket将方法所需要的参数传给B进程,然后B进程得到参数后执行,然后将结果通过socket再返回给A,看起来A就像能直接调用B中对象方法似的.
  6. 一个进程向另一个进程发送消息: 类似与一个对象向另一个对象发送消息,代码中表现为一个对象调用的了另一个对象的方法.
  7. 进程间通信: 在Android上下文可以看作远程过程调用.

Binder机制及概念模型

Binder就是一种方便进程间进行通信的机制,其作用就类似于socket,只是因为Android中存在大量的client-server设计,为了方便进程间的通信,所以才设计了此机制.

在该模型中,有以下几个实体的存在,Server, Client, ServiceManger, Binder驱动. 其可以与互联网进行类比: Server就是服务器, Client就是客户终端, ServiceManager是DNS服务器(Client为了找到Server,需要向ServiceManager进行查询), Binder驱动类似于路由器(其是实现数据从Client到Server间传输的关键,因为路由器在网络编程中是对程序员透明,程序员不应该考虑的,同理,Binder驱动也应该在Client-server开发中是透明的).

另外,Server需要向ServiceManager注册,在这个过程中Server就是Client,而ServiceManager就是Server.

Binder可以看成是Client与Server之间的粘合剂,或者可以看成两者之间的一种管道,其有两端, 在Server端是一个Binder实体,而在Client是一个对该实体的引用,也可以称为代理(Proxy), 当Server端有了实体并且Client有了代理之后,相当于在Server与Client打通了数据通道(其实现依赖于底层了Binder驱动),于是就可以进行远程过程调用了.

Binder模糊了进程的边界,当整个系统的各个Server与Client之间通过Binder粘合之后,整个系统似乎就是一个大的进程,各个Server与Client则是这个大进程中的一个个组件.

可知,Binder是一种简化进程间通信的概念模型.

编写系统Server的流程

参考文章

  1. 在Android系统中编写JNI方法在应用程序框架层提供Java接口访问硬件
  2. 在Android系统的应用程序框架层增加硬件服务接口
  3. 在Android系统中编写APP通过应用程序框架层访问硬件服务

具体流程

  1. 编写IXXXService.aidl定义所要实现的服务接口
  2. 根据IXXXService.aidl生成IXXXService.java,其中包含IXXXService.Stub用于Server端,包含IXXXService.Stub.Proxy用于Client访问服务的代理.
  3. 编写XXXService继承IXXXService.Stub,声明native api.
  4. 利用JNI和C/C++语言实现3中定义的native api,这里是真正实现Server功能的代码, 而Java只是对底层的Server, Client以及Binder机制进行包装而已.
  5. 将4中实现的Server注册到相应的文件中,这样,当Android系统初始化时,就包含了定义了JNI方法
  6. 在Java层,在SystemServer.java中通过ServiceManager.addService(new XXXService())来注册服务(之后就可以通过ServiceManager.getService来得到服务, 实现上得到的是一个Proxy)
  7. 重新打包system.img,这样在系统启动时就会自动加载XXXService了
  8. 利用ServiceManager.getService()得到服务(实际上得到的是BinderProxy对象)
  9. 利用IXXXService.Stub.asInterface()将8中得到的BinderProxy对象转换成IXXXService.Stub.Proxy对象,这是一个代理,可供Client端使用来访问服务
  10. 因为IXXXService.Stub.Proxy实现了IXXXService接口,所以可以利用该对象请求服务.

由图2可知IXXXService是由IXXXService.aidl生成的接口,实现该接口的一共有两个类,一个是IXXXService.Stub.Proxy,从名字可以知道,这是一个代理类,是Client访问Service的入口, 另一个是IXXXService.Stub,由具体流程的3, 4步可知,该类将由XXXService继承,并利用JNI来具体实现服务的代码,所以该类是服务端所使用的类.另外这两个类都与一个IBinder相关联,IXXXService.Stub.Proxy是包含一个IBinder成员,而IXXXService.Stub是继承自Binder,其本身就是一个IBinder.

那么为什么都两个类都要和一个IBinder相关联呢? 因为虽然这两个类一个供Client使用,一个供Server使用,这只是上层的抽象,但是要想使Client与Server进行通信,必须要有底层的实现,即使用某种进程间通信的机制,而由以上可知,这种通信机制就是Binder机制,所以Proxy和Stub都与一个Binder相关联就是为了使Proxy与Stub利用Binder机制进行通信.

其中Proxy关联的IBinder就是一个BinderProxy(从名字可以看出其用在Client端), 而Stub继承自Binder,所以使用的就是Binder. IBinder, Binder, BinderProxy的关系如图一所示.

因为Proxy和Stub都关联着一个IBinder对象,这里就可以解释IInterface了,该接口仅有的函数就是返回Proxy和Stub中关联的IBinder对象

final关键字与匿名内部类

发表于 2016-01-20

java中匿名内部类要使用包围它的函数的形参或局部变量时,需要声明其为final的,为什么?

这里有一个前提,就是当匿名内部类要使用包围它的函数的形参或局部变量时,javac会为该内部类自动一个构造函数,该构造函数拷贝一个形参或局部变量的副本.也就是说,在匿名内部类中使用的变量实际不是原来的变量了,只是副本.

设想,如果不需要声明成final的,那么,对于同一个变量名,实际上有两个,一个是在匿名内部类使用,一个是在函数中使用,对这两个变量的使用是独立的,对于程序员来说,看到同一名字的变量有不同的值会感到困惑(如程序员想在内部类内修改局部变量的值, 但因为修改的是副本,所以并不能修改成功),所以java要求将形参或局部变量声明为final的,并且javac会为匿名内部类声明一个同名的变量,也是final,并且使用自动生成的构造函数将外部的值付给内部的变量,这样,无论是外部,还是内部,都是final,都是不可变的,所以不会造成困惑.

由此可知,需要声明为final的,完全是因为自动生成的构造函数拷贝了一份,为了防止不一致的措施罢了.那么,为什么java要设计成自动生成的构造函数拷贝一份呢?

7. phpstorm type hint,laravel-ide-helper以及生成的文件,laravel plugin

phpstorm利用phpdoc进行代码提示与自动完成功能.phpdoc就是为类,函数,变量等添加一些元信息,phpstorm读取这些元信息就可以进行代码提示.

而laravel facade是对某一个实际类的包装,而phpstorm无法解析这种包装,所以对于laravel facade不能进行代码提示.

laravel-ide-helper就是分析laravel facade,找到其实际所指向的类,并为该类包装一个同名类,包装该类的所有静态方法,并添加phpdoc,这就是生成的-ide-helper.php文件.phpstorm读取该文件中的phpdoc,也就可以对facade进行代码提示了.

所以说laravel-ide-helper是为了弥补phpstorm对facade的解析失败

而laravel-plugin是phpstorm的一个插件,具体特性见文档

8. xdebug

xdebug的原理

利用c/s的架构. server端就是服务器上安装的xdebug,可以停止和运行php程序,并且获得程序的运行信息,而client端负责对server下达命令,命令其何时停止程序运行并返回信息.

client一般由ide自带,或通过插件的方式

工作流程如下

  1. 如果url中包括XDEBUG_SESSION参数,则启动xdebug server
  2. xdebug server启动后,根据http中的请求,或php.ini中的xdebug.remote_host来确定xdebug client的地址,并发起dbgp连接
  3. xdebug client必须开启监听(即phpstorm中的监听图标),当监听到xdebug server的连接后建立连接
  4. 连接建立后,xdebug client和server就可以通过dbgp协议来进行通信.xdebug client发生server断点信息等,server回传程序运行到断点时程序的状态
