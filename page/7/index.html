Parcelable与ClassLoader

发表于 2016-03-27

利用Messenger, 向远端Service传递自定义Parcelable对象时, 代码如下

自定义Parcelable类

public class Person implements Parcelable{}

客户端

Bundle bundle = new Bundle();
bundle.putParcelable("Test", new Person("dongpuo" , "su"));
msg.setData(bundle);
mService.send(msg);

服务端

Bundle bundle = msg.getData();
Person person = bundle.getParcelable("Test");

在运行到bundle.getParcelable("Test")时会报如下错误

android.os.BadParcelableException: ClassNotFoundException when unmarshalling: io.github.lingnanlu.clientserverdemo.Person

即当unmarshalling时, 找不到Person类, 这是什么原因呢?

找不到Person类, 说明Person.class对象没有生成, 而Person.class对象又是由ClassLoader生成的, 所以这样牵扯到了Android中的ClassLoader机制.

经过Google, 了解到Android的ClassLoader机制大概如下

Android has two different classloaders: the framework classloader (which knows how to load Android classes) and the APK classloader (which knows how to load your code). The APK classloader has the framework classloader set as its parent, meaning it can also load Android classes.

即, 有两类Classloader, 一类是framework classloader(BootClassLoader), 该classloader主要是负责加载framework中的类, 另一类是APK classLoader(具体名称是PathClassLoader, 下面会介绍怎么知道的), 负责加载APK中定义的类.

查看Bundle的源代码, Bundle中包含着一个ClassLoader, 当调用Bundle的get方法时, 会利用该ClassLoader加载需要unmarshalling的类, 进而再组装成对象.

由于可以推断, 通过msg.getData()取出来的bundle中的ClassLoader, 并不是APK classLoader, 所以无法加载Person.

我们可以通过输出ClassLoader来进一步观察. 修改以上代码如下

客户端

Bundle bundle = new Bundle();
Log.i(TAG, bundle.getClassLoader());
Log.i(TAG, Message.class.getClassLoader());
Log.i(TAG, Bundle.class.getClassLoader());
bundle.putParcelable("Test", new Person("dongpuo" , "su"));
msg.setData(bundle);
mService.send(msg);

服务端

Bundle bundle = msg.getData();
Person person = bundle.getParcelable("Test");
Log.i(TAG, bundle.getClassLoader());

运行程序后, 客户端打印出来的都是

java.lang.BootClassLoader

而服务端打印出来的为

null

以上可以说明Person类找不到的原因. 还说明Bundle和Message, 都是由BootClassLoader加载的, 因为这两个类是Framework类.

客户端的第三个Log也是输出java.lang.BootClassLoader, 可这里, 我们并没有为其显示的指定其ClassLoader, 查看基类的构造函数, 可以看出原因.

mClassLoader = loader == null ? getClass().getClassLoader() : loader;

我们知道, 跨越进程传输数据, 要不是基本类型, 要不就是Parceble, 在这里Message与Bundle对象也是跨进程进行传输的, 查看这两个类定义的文件, 发现这两个类都是Parceble的, 又因为这两个类的ClassLoader是BootClassLoader, 所以在服务器端组装这两个类对象没有问题.

那么为什么bundle本来有一个BootClassLoader, 在传输到了服务端之后就为null了呢?

我猜想是这样的.

ClassLoader并不是一个Parceble, 所以无法传输到服务端, 所以为null.

现在好了, 知道原因了, 因为bundle缺乏PathClassLoader, 所以无法加载类, 解决方法就是在getParcelable之前, 显示的设置PathClassLoader. 如

bundle.setClassLoader(自定义的一个类.class.getClassLoader());

注, ClassLoader加载类的机制是首先委托给父ClassLoader加载, 如果父ClassLoader加载不了, 再由子ClassLoader加载. 这样的限制避免了用户自己的代码冒充核心类库的类访问核心类库包可见成员的情况。这也好理解，一些系统层级的类会在系统初始化的时候被加载，比如java.lang.String，如果在一个应用里面能够简单地用自定义的String类把这个系统的String类给替换掉，那将会有严重的安全问题。

  2. Unmarshalling errors in Android app with custom parcelable classes

Service实现客户端-服务器模型的三种方式

发表于 2016-03-27

方式一: Extending the Binder class

这种方式也叫做Local Service, 因为该Service与客户端运行在同一个进程当中(可能不是同一线程), 所以只有本应用内的客户端可以使用该Service

适用情况:

如一个音乐App, 利用一个Service在Background播放音乐, 一个Activity与该Service绑定来控制音乐的播放与暂停.

可传输的数据类型:

因为在同一个进程, 所以无论是简单数据还是自定义的类, 都不需要Parcelable

方式二: Using A Messenger

这是最简单的方式来实现IPC, 客户端与服务端运行在不同的进程当中, 而且服务端得到的请求是进行了排队, 所以编写服务端时不需要进行线程安全的处理.

其实上, 查看Messenger的源码, Messenger只是对AIDL进行了一个封装, 关于Messenger的实现, 参考以下文章

Android应用进程间通信之Messenger信使使用及源码浅析

可传输的数据类型:

对于简单数据可以传输, 对于自定义的类, 因为是跨进程通信, 需要为Parcelable类型.

因为是用Message来传输, 典型用法如下:

Message msg = Message.obtain(null, MessengerService.MSG_SAY_HELLO, 0, 0);
Bundle bundle = new Bundle();
bundle.putParcelable("Test", new Person("kobe", "bryant"));
msg.setData(bundle);
mService.send(msg);

方式三: Using AIDL

最灵活与最复杂的方式, Messenger的底层实现, 需要自己来处理服务端的线程安全.一般很少用.

可传输的数据类型:

对于简单数据可以传输, 对于自定义的类, 因为是跨进程通信, 需要Parcelable.

补充知识点 Parcelable VS Serializable

两者的目的不同, Serializable是为持久化, 可以写到外存, 数据库, 网络等.

Parcelable是为了在不同进程中传输对象, 不是为了持久化, 只发生在内存中.

  2. Android应用进程间通信之Messenger信使使用及源码浅析
  3. Bound Services

使用sqlite需要主要关注的两个方面

发表于 2016-03-26

一 数据的CRUD操作以及相应的抽象

目的, 使使用者使用起来更方便, 不容易出错, 所以需要封装一些细节的,容易出错的东西

为了达到以上目的, 一般会设计如下几个类

  1. Contract类
  2. SQLiteOpenHelper类
  3. SQLiteDatabase类
  4. 封装SQL语句的类

以下说明这三种类的存在必要性, 所扮演的角色, 以及三者之间的关系

1. SQLiteOpenHelper

这是一个helper类, 主要是帮助数据库的创建和管理数据库的版本, 至于什么是数据库的版本, 下面会说明.

一般这个类的用法如下

public class CustomDBHelper extends SQLiteOpenHelper() {

    public void onCreate(SQLiteDatabase db) {
        // 利用db创建相应的表
    }

    public void onUpgrade(SQliteDatabase db, int oldVersion, int newVersion){
        //更新数据库版本时所做操作
    }
}

//以下是客户端代码
SQLiteDatabase db = new CustomDBHelper().getReadableDatabase();

一个数据库对应于一个SQLiteOpenHelper, 该类负责该数据库中所有表的创建与版本管理,该类的一个对象也关联着一个SQLiteDatabase对象, 而SQLiteDatabase负责执行具体的SQL语句.

2. SQLiteDatabase

该类负责执行具体的SQL语句, 和CRUD操作.由SQLiteOpenHelper得到该对象后, 这样使用:

//执行SQL语句
db.execSQL(String sql)

//CRUD
db.delete(String table, String whereClause, String[] whereArgs)
db.insert(String table, String nullColumnHack, ContentValues values)
db.query()
db.update(String table, ContentValues values, String whereClause, String[] whereArgs)

//其它常用
db.beginTransaction()
db.endTransaction()

3. 封装SQL语句的类

在程序中, 如果直接使用SQL语句来做CRUD操作, 有如下弊端

  1. 很不方便, 每次操作都需要写很大的SQL语句
  2. 容易出错, SQL语句就是一个字符串, 格式很严格, 少一个空格, 多一个符号都不行, 如果多处使用, 就存在多处出错的可能性
  3. 不够直观, 因为使用数据库操作时, 往往有一个实体对象与之对应, 直接使用SQL语句操作实体对象并不直观

可以通过对实体的常用操作封装一个类, 将真实的SQL语句放在类接口的实现中, 这样, 就可以一定程度上缓解以上问题.

但还是有一点问题, 需要为每一个实体封装一个类, 而该类的接口往往都是相似的, 违反了DRY原则, 所以更好的解决方案是使用ORM.

4. Contract类

该类并非必要, 只是提倡有这样一个类, 为的是程序的易读性和不容易出错.
往往一个数据库对应一个Contract类, 数据库中的一个表对应该类的一个静态内部类, 以Google官方的例子来说.

public final class FeedReaderContract {

    //每一个表
    public static final class FeedEntry implements BaseColumns {

        //该Table的描述
        public static final String TABLE_NAME = "entry";
        public static final String COLUMN_ENTRY_ID = "entryid";
        public static final String COLUMN_TITLE = "title";
        public static final String COLUMN_CONTENT = "content";
        public static final String COLUMN_NULLABLE = "";

        public static final String SQL_DELETE_ENTRIES =
            "drop table if exists " + TABLE_NAME;

    }

}

为一些SQL字段与语句定义一些常量, 在程序中使用常量而不直接使用对应的字符串, 可以使其更易读和不易出错.

二 数据库的版本管理

数据库的版本指的是什么?

假如当前有一个数据库, 库中有一个表, 将当前数据库的状态当做版本0

如果后来将该表的schema更改了, 此时可以认为数据库的版本改变了, 成为一个新的版本.

什么是数据库的版本管理

数据库的版本管理是指,当数据库的schema更改时.

  1. 如何将之前的schema改变为新的schema
  2. 如何将之前的数据Migrate到新的表中

关于版本管理的两个方法

public abstract class SQLiteOpenHelper {

    public abstract void onCreate(SQLiteDatabase db);
    public abstract void onUpgrade(SQLiteDatabase db, int oldVersion, int newVersion);

}

基中, onCreate是在数据库第一次创建时调用的. 而onUpgrade是数据库的版本号不同时进行调用的, 而数据库的版本号是在创建SQLiteOpenHelper的对象时, 传入到构造函数中, 并且保存在数据库中的.该函数中的实现需要做到不同版本间数据的migration

一个实例

2013年4月，我们第一次发布了我们的应用，数据库版本是1。

2013年5月，我们第二次发布了我们的应用，数据库版本是2。由于业务需要，我们更改了数据库里的某个表的表结构。

这时候就有这样的难题出现：

有些用户已经下载了4月份的版本1，并且已经使用了，很多数据存储在数据库了，这个时候，他想安装新的版本2，怎么办？ 怎么才能让数据不丢失？

这个时候就需要使用onUpgrade的方法了.

开发过程中表的schema需要改变怎么办?

开发过程中表的schema也可能是经常变的, 怎么去做版本的管理?

有如下两种方法

  1. 手动删除db文件, 这样每次运行时, 会创建新的db文件
  2. 每更新一次schema就改变一次版本, 并在onUpgrade中删除旧表, 重新创建新表

表中的数据怎么办?

这个问题其实不用担心, 因为是开发过程中, 所有的数据应该是test数据, 并且是使用程序自动填充的, 所以不需要考虑数据的保留问题
